Time Complexity

First it is worth mentioning that the program runs in below 2 seconds for inputs as large as 1000 lines of proof on a usual Intel i5 core processor.

Having said that the timing mentioned in the following table is the sum of the times taken by all cores combined (which run simultaneously) and hence is much more than 2 seconds for inputs of size around 1000.

In a way it is the time if a single processor completed the entire process at a go.

Complexity as length of a given line increases.

The timing of the code is unaffected by the length of a particular proof line since the total number of comparisons made remain the same irrespective of the number of levels in the tree of a particular proof line.

Complexity as number of lines of proof increases

Now we know that the length of a particular line, doesn't effect our complexity.

Also in a particular line only a single rule can be applied.

So each line will be of the form:

  1. (t/^e1/line1) or (t/^e2/line1)
  2. (tVs)/Vi1/line1/line2 or (tVs)/Vi2/line1/line2
  3. (t^s)/^i/line1/line2
  4. t/>e/line1/line2
  5. (~t)/MT/line1/line2

Note: here t and s can themselves be literals or literals connected by operators.

Now, the first 2 in the list(^e and Vi) need to match all elements of 2 rows in the 2D array. While the other three need to compare 2 pairs of rows in the array in the worst case.

Therefore we can take one rule from the first set and one rule from the second set and test each one individually for around 1000 inputs. We can then average the times.
